现代C++中的左值和右值及其作用

在早期C/C++中，没有移动语义，也没有引用 collapsing。那时候的定义比较粗糙：左值被视为指代一个对象的表达式，而右值则被视为了所有并非左值的东西，或者说“表达式的值”。在很多古老的教材或非严格的技术讨论中，人们会混用这两个概念（即值value和右值rvalue）：

“计算 a + b，我们会得到一个值（value）。”

有时候他们会把这个“值”直接称为 “rvalue”。

在这个宽泛的、不严谨的语境下，你可以这样理解那句话：

“所有表达式最终都会产生一个值（Value），所以所有左值也都是一个值（他们称之为 Rvalue）。”

但这不是现代 C++ 编程中使用的定义。

C++17 标准对表达式值类别的定义如下：

  • glvalue 是一个表达式，它的计算可以确定对象、位域或函数的标识。
  • prvalue 是一个表达式，它的计算可以初始化对象或位域，或计算运算符的操作数值，这是由它出现的上下文所指定的。
  • xvalue 是一个 glvalue，表示一个对象或位域，该对象或位域的资源可重复使用（通常是因为它接近其生存期的末尾）。 示例：某些涉及 rvalue 引用 (8.3.2) 的类型的表达式会生成 xvalue，例如对返回类型为 rvalue 引用或强制转换为 rvalue 引用类型的函数的调用。
  • lvalue 为非 xvalue 的 glvalue。
  • rvalue 是一个 prvalue 或 xvalue。

关系如下所展示

lvalue 具有程序可访问的地址。 例如，lvalue 表达式包括变量名称，其中包括 const 变量、数组元素、返回 lvalue 引用的函数调用、位域、联合和类成员。

prvalue 表达式没有可供程序访问的地址。 例如，prvalue 表达式包括文本、可返回非引用类型的函数调用，以及在表达式计算期间创建的但只能由编译器访问的临时对象。

xvalue 表达式具有程序不再可访问的地址，但可用于初始化右值引用，该引用提供对表达式的访问权限。 例如，它包括可返回 rvalue 引用的函数调用，以及数组下标、成员和指向其中数组或对象是 rvalue 引用的成员表达式的指针。

上面这段引用读起来或许会让人一头雾水，让我们先举些生动的例子来让我们理解。

  • 首先是左值lvalue，它是我们最常接触的且易于理解的，例如int a = 10;中a便是lvalue,它有一个地址，我们可以使用一个指针绑住它int *p = &a;，同时其像一个盒子，我们可以使用a = 2;来修改其中内容。

  • 而纯右值prvalue则就是一个幽灵，就像int c = a + b;中的a+b其一旦被计算得到，便会赋值给c，再也不会存在，一般来看，纯右值要么会被编译期优化为立即数或者仅存在在寄存器中，如果是较大的复杂对象，编译器会将其临时存在一个程序员不可见的栈上；

  • xvalue（将亡值）是一种特殊的 glvalue，它标识了一个对象接近其生存期末尾，其资源可以被重用。它通常通过返回右值引用的函数或 std::move 产生，可以将左值修改为右值引用，表示我不再需要这个对象的当前值，可以安全地将其资源转移。它是现代 C++ 高效资源管理的核心工具之一，尤其在容器操作、类设计和性能优化中至关重要。可以实现移动而非拷贝。例如

    template <typename T>

void mySwap(T& a, T& b) {
    T tmp = std::move(a); // 移动构造
    a = std::move(b);     // 移动赋值
    b = std::move(tmp);   // 移动赋值
}

  • glvalue即是我们说的可以被取地址的值

  • rvalue则是是一个 prvalue 或 xvalue。

值类别的工程意义：移动语义与性能优化

在 C++11 引入移动语义之前，值类别的区分主要用于语法层面的校正。然而，现代 C++ 将这一概念提升到了性能优化的核心地位，主要体现在避免不必要的深拷贝。

1. 深拷贝的性能开销

考虑以下场景：一个函数返回一个包含大量数据的对象，如 std::vector。

std::vector<int> createVector() {
    std::vector<int> temp(1000000); // 构造包含 100 万元素的向量
    return temp;
}

int main() {
    std::vector<int> v = createVector(); 
}

在旧的 C++ 标准（C++11 之前）中，如果没有编译器的优化（如 RVO），上述代码会发生深拷贝：

  1. 函数内的局部对象 temp 被拷贝给 main 函数中的临时对象。
  2. 临时对象再被拷贝给 v。
  3. 每次拷贝都需要分配内存并复制一百万个整数，产生巨大的 CPU 和内存开销。

2. 移动语义的引入

现代 C++ 利用值类别来区分“拷贝”与“移动”：

  • 左值： 代表持久存在的对象。对其进行赋值通常意味着“拷贝”，因为源对象后续可能仍需使用，必须保持原状。
  • 右值： 代表临时的或即将销毁的对象。对其进行赋值意味着“移动”，因为源对象即将消亡，窃取其内部资源（如堆内存指针）不仅安全，而且高效。

std::move 并不移动任何东西，它本质上是一个静态类型转换，将一个左值强制转换为xvalue（将亡值）。这告诉编译器：“该对象不再被需要，可以安全地调用移动构造函数或移动赋值运算符”。

优化后的 std::swap 实现：

template <typename T>
void mySwap(T& a, T& b) {
    T tmp = std::move(a); // a 转换为 xvalue，tmp 接管 a 的资源
    a = std::move(b);     // b 转换为 xvalue，a 接管 b 的资源
    b = std::move(tmp);   // tmp 转换为 xvalue，b 接管 tmp 的资源
}

在该实现中，对于像 std::string 或 std::vector 这样的对象，整个过程仅涉及指针的交换，无需任何内存数据的复制，时间复杂度从 O(N) 降低至 O(1)。

Rust 中的值类别与所有权机制

Rust 借鉴了 C++ 的值类别概念，并将其整合到其核心的所有权和借用系统中。虽然 Rust 的术语与 C++ 略有不同，但其底层逻辑依然基于表达式的生命周期和内存位置。

1. 移动作为默认行为

在 C++ 中，左值的默认行为是拷贝，移动通常需要显式地使用 std::move 触发。而在 Rust 中，对于未实现 Copy trait 的类型（如 String, Vec），赋值操作默认即为移动。

let s1 = String::from("hello");
let s2 = s1; 
// println!("{}", s1); // 编译错误：use of moved value: `s1`

尽管 s1 在语法上是一个左值（具有内存地址），但在赋值给 s2 的语义中，编译器将其视为资源可被转移的实体。一旦所有权转移，原变量 s1 即被失效，这从编译器层面杜绝了“悬垂指针”的风险。

2. 生命周期与借用

Rust 的借用检查器严格依赖左值和右值的生命周期属性：

  • 左值： 拥有明确的生命周期作用域，可以通过引用进行共享或可变借用。
  • 右值： 通常是表达式计算产生的临时值。Rust 规定，临时值的生命周期通常不能超出包含它的语句，除非通过 let 绑定将其延长。

这种严格的机制对应了 C++ 中“不可对右值取地址”的底层约束，但 Rust 将其上升为语法规则，从而在无需垃圾回收器的情况下保证内存安全。

总结

现代系统级编程语言对左值和右值的区分，早已超越了简单的“赋值位置”描述。实际上，这是对对象生命周期和资源管理权限的精确划分：

  1. 身份与持久性：

    • 左值 标识了持久存在的对象，可以安全地取地址并长期持有引用。
    • 右值 标识了临时或即将销毁的对象，其资源可以被重用。

  2. 性能与安全：

    • C++ 利用右值属性实现移动语义，消除了不必要的深拷贝，极大提升了程序性能。
    • Rust 利用所有权移动规则，在编译期强制执行内存安全，防止了数据竞争和内存泄漏。

理解这些概念，是深入掌握现代 C++ 高性能编程和 Rust 内存安全模型的关键基础。
